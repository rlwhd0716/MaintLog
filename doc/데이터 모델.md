# 📄 DATA_MODEL.md

---

## 🧭 1. 개요

본 프로젝트는 유지보수 업무에서 발생하는 요청사항, 장애, 작업 이력을 관리하기 위한 데이터 구조를 정의한다.

핵심 설계 방향:

- 상태 기반 관리 구조
- 작업 이력(로그) 중심 설계
- 확장 가능한 도메인 구조

---

## 🧱 2. 주요 엔티티

### 2.1 Request (요청사항)

| 필드명 | 타입 | 설명 |
|--------|------|------|
| id | Long (PK) | 요청 ID |
| title | String | 요청 제목 |
| description | String | 요청 내용 |
| requester | String | 요청자 |
| status | String | 상태 (등록, 진행중, 완료 등) |
| priority | String | 우선순위 |
| createdAt | Long | 생성 시간 |
| updatedAt | Long | 수정 시간 |

---

### 2.2 RequestLog (작업 로그)

| 필드명 | 타입 | 설명 |
|--------|------|------|
| id | Long (PK) | 로그 ID |
| requestId | Long (FK) | 요청 ID |
| content | String | 작업 내용 |
| author | String | 작성자 |
| status | String | 변경 상태 |
| createdAt | Long | 생성 시간 |

---

### 2.3 Incident (장애)

| 필드명 | 타입 | 설명 |
|--------|------|------|
| id | Long (PK) | 장애 ID |
| title | String | 장애 제목 |
| description | String | 장애 내용 |
| status | String | 상태 (발생, 조치중, 해결 등) |
| severity | String | 중요도 |
| createdAt | Long | 발생 시간 |
| resolvedAt | Long? | 해결 시간 |

---

### 2.4 IncidentLog (장애 조치 로그)

| 필드명 | 타입 | 설명 |
|--------|------|------|
| id | Long (PK) | 로그 ID |
| incidentId | Long (FK) | 장애 ID |
| content | String | 조치 내용 |
| author | String | 작성자 |
| createdAt | Long | 생성 시간 |

---

## 🔗 3. 관계 (Relationship)

- Request (1) : RequestLog (N)
- Incident (1) : IncidentLog (N)

---

## 🧩 4. 상태 정의

### 4.1 Request Status

- REGISTERED (등록)
- IN_PROGRESS (진행중)
- COMPLETED (완료)
- HOLD (보류)
- CANCELED (취소)

---

### 4.2 Incident Status

- OPEN (발생)
- IN_PROGRESS (조치중)
- RESOLVED (해결)
- CLOSED (종료)

---

## 🛠 5. 설계 포인트

### 5.1 상태 기반 구조
- 모든 업무는 상태를 통해 관리된다.

### 5.2 로그 중심 설계
- 작업 과정은 로그로 누적된다.
- 상태 변경도 로그로 기록 가능하다.

### 5.3 확장 고려
- Attachment 테이블 추가 가능
- User 테이블 확장 가능
- API 연동 구조 대응 가능

---

## 🚀 6. 향후 확장

- Attachment (파일 첨부)
- User (사용자 관리)
- Notification (알림)
- History (변경 이력 추적)

---

## 📌 7. 한 줄 요약
